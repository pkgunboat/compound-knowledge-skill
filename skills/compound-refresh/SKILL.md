# /compound-refresh — 知识库维护

## 目标

审查 `~/.claude/knowledge/solutions/` 中的文档，确保知识库保持准确、精简、可用。

## 执行流程

### Phase 1: 扫描知识库

1. 用 Glob 列出 `~/.claude/knowledge/solutions/**/*.md` 所有文档
2. 读取每份文档的 frontmatter（title、category、tags、created、last_updated）
3. 按 category 分组，统计总量

输出概览：
```
知识库概览:
  总文档数: N
  分类分布: [各分类及数量]
  最旧文档: [文件名] ([日期])
  最新文档: [文件名] ([日期])
```

### Phase 2: 逐份评估

对每份文档执行以下检查：

1. **引用有效性** — 文档中提到的文件路径、函数名、命令是否仍然存在？（用 Grep/Glob 验证）
2. **方案时效性** — 解决方案是否因技术栈升级而过时？
3. **冗余检测** — 是否有多份文档描述同一问题的不同变体？

### Phase 3: 分类判定

| 判定 | 条件 | 操作 |
|------|------|------|
| **保留 (Keep)** | 内容准确，无需修改 | 不动 |
| **更新 (Update)** | 核心方案有效，但引用的路径/类名/API 已变 | 更新引用，刷新 `last_updated` |
| **合并 (Consolidate)** | 两份或多份文档高度重叠 | 合并为一份，保留所有独特内容，删除被合并的文档 |
| **替换 (Replace)** | 核心方案已不正确或有更好方案 | 重写文档内容 |
| **删除 (Delete)** | 问题领域已不存在，或完全冗余 | 删除文件 |

### Phase 4: 用户确认

列出所有非 "保留" 的操作计划，格式：

```
维护计划:
  更新 (N):
    - [文件路径]: [变更摘要]
  合并 (N):
    - [文件A] + [文件B] → [目标文件]: [原因]
  替换 (N):
    - [文件路径]: [旧方案] → [新方案]
  删除 (N):
    - [文件路径]: [原因]
```

**等待用户确认后再执行操作。**

### Phase 5: 执行操作

按确认的计划逐一执行：
- 更新：用 Edit 修改
- 合并：用 Write 写入合并后文档，删除被合并的文档
- 替换：用 Write 重写
- 删除：用 Bash `rm` 删除

### Phase 6: 输出报告

```
维护完成:
  保留: N 份
  更新: N 份
  合并: N 份 (减少 M 份)
  替换: N 份
  删除: N 份
  当前总量: N 份
```

## 关键原则

1. **匹配现实，而非反过来** — 如果代码变了，更新文档而不是忽略差异
2. **合并优先于删除** — 删除前必须确认独特内容已被保留
3. **替换而非删除过时方案** — 除非问题领域本身消失了，否则用新方案替换旧方案
4. **最小化改动** — Keep 优于 cosmetic edit；不要为了美观做无意义修改
5. **保守操作** — 有疑问时保留文档，标记为待确认
